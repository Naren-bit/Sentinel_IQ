# SentinelIQ — Project Writeup

## What We Built

SentinelIQ is an AI-powered PII review prioritization tool designed to solve a specific, high-stakes problem: when an AI flags dozens of personally identifiable information (PII) items in a document, **which ones should a human reviewer look at first?** Rather than building yet another PII detector, we built the layer that sits *on top* of detection — a priority engine that ranks every flag by how likely it is to be missed by a human reviewer.

The system follows a three-stage pipeline: **Detect → Verify → Prioritize**. Documents are analyzed by a primary AI provider (Google Gemini) which identifies PII entities with character-level offsets. A deterministic verification pass then sweeps for candidates the AI may have missed — phone-number-shaped substrings, repeated proper nouns with only partial detection coverage. Finally, a six-rule heuristic engine scores every detection on factors like non-canonical formatting (the heaviest signal, because unusual formats defeat both AI pattern-matching *and* human scanning), late document position (reviewer fatigue), single occurrence (no redundancy to catch a miss), surrounding text density, low AI confidence on sensitive types, and consequence-if-missed by PII category. Detections are bucketed into HIGH, STANDARD, and LOW priority tiers with human-readable explanations for every score.

The frontend is a Next.js application with a premium glassmorphism UI built around a dynamic liquid gradient background. It presents an Attention Queue — a scrollable list of glass-effect detection cards, each showing the flagged text in context, the reasons it was prioritized, and a large confidence score. Reviewers can approve or dismiss flags individually, with the document viewer highlighting each entity inline. The entire architecture uses a provider abstraction with graceful fallback: if the cloud AI times out or fails, the system silently switches to an offline mock engine, ensuring the demo never breaks.

## What We Intentionally Did Not Build

We excluded **authentication and user management** — login flows, JWTs, and role-based access are solved problems that would consume time without demonstrating our core innovation. We excluded **database persistence** — storing PII at rest demands encryption, lifecycle management, and compliance overhead that is orthogonal to the prioritization concept; instead, all state lives in-memory via Zustand on the client. We excluded **batch processing** (uploading many documents at once would require task queues, WebSockets, and UI virtualization) and a **custom heuristics configuration UI** (letting users tune rule weights is a V2 enterprise feature, not a hackathon deliverable). Finally, we deliberately chose a **deterministic heuristic engine over a trained ML ranking model** — a black-box model cannot explain *why* a detection is high priority, but our rules produce exact, human-readable justifications like *"Non-standard phone format — unusual formatting defeats visual scanning"*, which is critical for trust in regulated industries.
