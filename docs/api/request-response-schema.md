# JSON Schema Documentation

This document describes the core JSON objects passed between the SentinelIQ Backend and Frontend.

## 1. Detection
The raw entity object extracted by the LLM (or Mock Provider).

```json
{
  "id": "string",            // Unique identifier (e.g., "det-001")
  "text": "string",          // The exact character substring found in the document
  "start": "integer",        // 0-indexed character offset where the text begins
  "end": "integer",          // 0-indexed character offset where the text ends (exclusive)
  "type": "string",          // Enum: NAME, PHONE, EMAIL, ADDRESS, ID, OTHER
  "confidence": "number"     // Float between 0.0 and 1.0 indicating AI certainty
}
```

## 2. ReviewPriority (Enrichment)
The metadata generated by the `ReviewPriorityEngine` describing how urgent the detection is.

```json
{
  "detectionId": "string",   // Foreign key matching Detection.id
  "priorityTier": "string",  // Enum: HIGH, STANDARD, LOW
  "reasons": [               // Array of human-readable justifications
    "string"                 
  ],
  "drivingSignal": "string"  // Enum: confidence, position, density, repetition, pattern, consequence
}
```

## 3. Reasons
Reasons are string literals generated by the heuristic rules. Examples include:
*   `"Only mention in the document — no second reference to catch if missed"`
*   `"Buried in a dense paragraph (311 chars) — easy to skim past"`
*   `"Low AI confidence (58%) on a ID — needs human verification"`

## 4. VerificationStatus (Frontend State)
The status of verification and offsets repair.
*Note: In SentinelIQ, this is handled via backend logging and the `fallbackOccurred` boolean flag sent to the client, rather than a dedicated nested object.*

```json
{
  "fallbackOccurred": "boolean" // true if Gemini failed and MockProvider was used
}
```

## 5. ReviewDecision (Frontend State)
Managed entirely in the browser via Zustand `useStore`. We maintain a `Set<string>` of `reviewedIds`.

```typescript
// Zustand Store State
reviewedIds: Set<string> // Contains the detectionId of all items the user has clicked "Approve" on.
```

## 6. FinalReport (Conceptual)
When the user clicks "Bulk Approve" or finishes the queue, the session is cleared. In a production setting with a database, this would construct a payload containing the document ID, the list of approved redactions, and a reviewer timestamp. Currently, it triggers the frontend `Complete.tsx` success screen.
